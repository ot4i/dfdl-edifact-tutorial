Overview

This tutorial demonstrates the modeling of UN/EDIFACT data using DFDL schema. The DFDL schema editor is used to view the model and test parse example EDIFACT data files.

UN/EDIFACT data streams have a hierarchical structure where the top level is referred to as an interchange, and lower levels contain multiple messages which consist of segments, which in turn consist of composites and elements. Composites in turn consist of elements. Segments, composites and elements are delimited by separators. Only segments have initiators.

An interchange starts with a UNB segment and ends with a UNZ segment. A message starts with a UNH segment and ends with a UNT segment, an element in the UNH segment identifying the message type. Messages may be grouped into functional groups, which start with a UNG segment and end with a UNE segment. Optionally the interchange can start with a UNA segment which defines the delimiters used by the interchange; defaults are used if there is no UNA.

UN/EDIFACT data streams adopt one of several syntax levels, the most common being version 3 or version 4.

Import projects

Two shared libraries are imported into your workspace.

Library EDIFACT-D03B contains a pair of DFDL schemas that model a UN/EDIFACT Supply Chain interchange for release D.03B. Message types INVOIC and ORDERS are provided. Test data files are included in the library.

Library EDIFACT-Common contains a DFDL schema to define default values for DFDL properties, and a DFDL schema to model the Uxx service segments and service messages for syntax version 4.

Import

Imported projects

The EDIFACT-D03B and EDIFACT-Common libraries are shown in the Integration Development view of your workspace.

DFDL schema EDIFACT-SupplyChain-Messages-D.03B.xsd in the EDIFACT-D03B library should be open for you using the DFDL editor. If it is not, double-click the DFDL schema and the editor will open.

The Interchange message is highlighted and models a whole EDIFACT interchange. The main editor view shows the logical components of the message such as elements and sequences. You can explore the Interchange message structure by expanding the elements.

The physical rendering of each logical component is described by the DFDL properties in the Representation Properties tab on the right. DFDL properties can either be specified locally on the component, or can be inherited from pre-defined sets of DFDL properties. Inherited properties have an icon shown next to them, and hovering over the icon reveals where the property is defined. In this schema, inherited properties are obtained from IBM_EDI_Format.xsd in the EDIFACT-Common library.

Because the settings of delimiters in an EDIFACT interchange can vary, the DFDL Terminator, Separator, Escape Character and Decimal Separator properties are set dynamically using DFDL expressions that refer to DFDL variables. The variables have default values and are overridden by the settings in a UNA segment if present. You can see this if you expand the UNA element in Interchange, select any of the child elements, and click on the Variables tab next to Representation Properties on the right.

You will be test parsing example EDIFACT data using the Interchange message. The test parsing takes place entirely within the DFDL editor, so there is no message flow in this tutorial and no Deploy step. Before test parsing, switch to the DFDL Test perspective using Window->Open Perspective and choose 'DFDL Test' (or use the toolbar icon ).

Tip: If this Tutorial Steps View disappears when you switch perspectives, use Window->Show View to add it back.

Follow these steps to complete the tutorial

EDIFACT-SupplyChain-Messages-D.03B.xsd is open in the DFDL editor. The Interchange message should be highlighted in green.

  1. Test parse an example data file:
    • Click the Test Parse Model button in the DFDL editor toolbar. The Test Parse Model dialog opens.
    • In the Message section, select Interchange.
    • In the Parser Input section, select 'Content from a data file' and click Browse.
    • Select file ORDERS_D.03B_Interchange.txt from EDIFACT-D03B and click OK.
    • Set the Encoding to ASCII.
    • Click OK. If asked to confirm switching to the DFDL Test perspective, click 'Yes'.
  2. The results of the test parse are displayed. You should see 'Parsing completed successfully' which can be dismissed.
  3. You can view the parsed data file in the DFDL Test - Parse view. The results of the parse can be viewed in the DFDL Test - Logical Instance view, as a tree or as XML. You can view a log of the parser actions in the DFDL Test - Trace view.
  4. Test serialize the logical instance that resulted from the parse:
    • Click the Test Serialize Model button in the DFDL editor toolbar. The Test Serialize Model dialog opens.
    • In the Serializer Input section, select 'Content from a DFDL Test - Logical Instance'.
    • Set the Encoding to ASCII.
    • Click OK.
  5. The results of the test serialize are displayed. You should see 'Serialization completed successfully' which can be dismissed.
  6. You can view the serialized data file in the DFDL Test - Serialize view. You can view a log of the serializer actions in the DFDL Test - Trace view.
  7. Repeat steps 1 through 6 for another example data file, INVOIC_D.03B_Interchange_with_UNA.txt.
  8. As an alternative to the toolbar buttons, you can also test from the DFDL Test - Parse and DFDL Test - Serialize views, using the Browse button to select a data source and the green play icon to run the DFDL parser or serializer.

The supplied DFDL schemas will parse UN/EDIFACT interchanges with or without new lines at the end of each segment, but will serialize interchanges without new lines. If you need to serialize with new lines, you can edit EDIFACT-Service-Segments-4.1.xsd in the EDIFACT-Common library and change the DFDL expression used to set DFDL variable 'SegmentTerm' to the commented-out value.

If you need to support UN/EDIFACT syntax version 3 instead of syntax version 4, you can edit IBM_EDI_FORMAT.xsd in the EDIFACT-Common library and change DFDL variable 'RepeatSep' to have default value '+' instead of '*'.

If you need to support ',' (comma) as the default decimal separator instead of '.' (period), you can edit IBM_EDI_FORMAT.xsd in the EDIFACT-Common library and change DFDL variable 'DecimalSep' to have default value ',' instead of '.'.

The two shared libraries can be deployed to an integration node for use by message flows.
